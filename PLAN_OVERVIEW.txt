================================================================================
                           KEY INSIGHTS SUMMARY
================================================================================

PROBLEM:
  graphify & pi-scope solve the same problem from different angles
  - graphify: slow, semantic (what connects to what)
  - pi-scope: fast, syntactic (symbol matching + deps)
  
OPPORTUNITY:
  Load graphify's graph.json as supplementary context source
  - Gain semantic awareness without slowdown
  - Reuse proven, published code
  - Zero breaking changes

SOLUTION: Adapter Pattern
  ✅ graphify outputs: graph.json (nodes, edges, confidence, communities)
  ✅ pi-scope loads graph at session start (~100ms one-time cost)
  ✅ Enables:
     • God node detection (rank central files higher)
     • Surprising connections (unexpected cross-module links)
     • Confidence scoring (EXTRACTED > INFERRED > AMBIGUOUS)
     • Community-based pruning (keep relevant context only)
     • Report guidance (key concepts + suggested questions)

IMPACT:
  • Token savings: maintain ~85% (add <5% graph overhead)
  • Relevance: +15% (god nodes rank top-3 vs baseline top-5)
  • Breaking changes: 0 (all features optional, graceful fallback)
  • Compatibility: 100% (works without graph.json)

TIMELINE: 4-7 weeks (3+ developers) or 2-3 months (1 developer)
  Phase 1 (Weeks 1-3): Loader + retrieval scoring + surprises
  Phase 2 (Weeks 3-5): God nodes + LSP enhancement
  Phase 3 (Weeks 5-7): Community pruning + report guidance
  Phase 4 (Post-MVP): Semantic search, cross-repo, GraphQL API

================================================================================
                          KEY DECISIONS MADE
================================================================================

Decision 1: Adapter vs. Merge
  ✅ DECIDED: Adapter Pattern
  Why: graphify is standalone tool, pi-scope is optional consumer
       No code merging, no tight coupling, backwards compatible

Decision 2: Load timing
  ✅ DECIDED: Session start (once per session)
  Why: 100ms one-time cost, simplicity, easy to disable

Decision 3: MVP scope
  ✅ DECIDED: Phase 1-3 (loader + surprises + god nodes)
  Why: Delivers value, realistic timeline, foundation for Phase 4

Decision 4: Fallback behavior
  ✅ DECIDED: Graceful no-op if graph missing
  Why: Zero breaking changes, backwards compatible

Decision 5: Token budget
  ✅ DECIDED: ~2K max for graph features (within 8K injection budget)
  Why: <5% overhead for +15% relevance gain
